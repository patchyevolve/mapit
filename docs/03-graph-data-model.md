# Graph & Data Model Specification
## Project: `mapit`

**Document version:** 1.0
**Depends on:** 01-PRD.md, 02-TRD.md
**Audience:** AI coding agent (implementer)

This document is the single source of truth for what a "node" and "edge" mean in `mapit`. Every other component (parser output, SQLite schema, REST API responses, frontend rendering) must conform to these shapes. Do not invent parallel or divergent representations elsewhere in the codebase — if a new field is needed, this document is updated first, then code follows.

---

## 1. Node Types

Every node has these **common fields**, regardless of type:

```ts
interface BaseNode {
  id: string;                 // stable, content-derived ID (see §4)
  type: NodeType;              // see enum below
  name: string;                // display name, e.g. "parse_header", "e1000.c", "Networking"
  language?: string;            // e.g. "c", "rust", "asm", "python" — absent for non-code nodes (e.g. "feature")
  file_path?: string;           // relative to project root; absent for "feature" nodes
  span?: { start_line: number; end_line: number }; // absent for file/feature/module nodes
  ai_summary?: string | null;   // null = not yet generated; absent only if AI disabled entirely
  ai_summary_status: "pending" | "ready" | "unavailable";
  ai_model_used?: string | null; // which provider/model produced ai_summary, for provenance
  structural_hash: string;      // hash of the source span this node was derived from, used for incremental re-annotation (TRD §3.4)
  flaws: FlawFlag[];            // see §3, always present (empty array if none)
}

type NodeType =
  | "feature"      // AI-classified subsystem/feature grouping (no source span)
  | "module"        // language-level module/namespace/package, where the language has one
  | "file"
  | "function"      // includes methods, free functions, and (for asm) labeled routines
  | "type"           // struct/class/enum/union definition
  | "macro"
  | "global"         // global variable / constant
  | "external"       // a call/reference target that could not be resolved to project source (TRD §3.2 "dangling")
```

### 1.1 Type-specific extra fields

```ts
interface FeatureNode extends BaseNode {
  type: "feature";
  member_node_ids: string[];      // files/functions/etc. AI-assigned to this feature
  classification_confidence: number; // 0.0–1.0
}

interface FileNode extends BaseNode {
  type: "file";
  size_bytes: number;
  parse_status: "ok" | "parse_failed" | "unsupported_language";
  parse_error?: string;            // present only if parse_status == "parse_failed"
}

interface FunctionNode extends BaseNode {
  type: "function";
  signature: string;               // best-effort textual signature as written in source
  is_entry_point_candidate: boolean; // heuristic: main(), exported/pub fn, HTTP handler patterns, interrupt handler patterns, etc.
  has_incoming_calls: boolean;      // direct structural fact, used for dead-code reasoning (TRD §5.2)
  control_flow?: ControlFlowGraph;  // see §5; may be absent if not yet computed
}

interface ExternalNode extends BaseNode {
  type: "external";
  reason: "no_source_present" | "dynamic_dispatch" | "unrecognized_binding";
}
```

## 2. Edge Types

```ts
interface Edge {
  id: string;                  // stable hash of (from_id, to_id, type, order_hint)
  from_id: string;
  to_id: string;
  type: EdgeType;
  confidence: "exact" | "probable" | "dynamic_unresolved"; // TRD §3.2
  order_hint?: number;          // for "calls" edges: sequence position within the caller's body, used for execution-order reconstruction (§5)
  condition?: string | null;    // for "calls" edges reached only conditionally: a short human-readable condition string extracted from the branch, e.g. "if (status == ERROR)"; null if unconditional
}

type EdgeType =
  | "calls"          // function -> function
  | "includes"        // file -> file (#include, import, use, require)
  | "defines"          // file/module -> function/type/macro/global
  | "references"       // function -> type/global/macro (non-call usage)
  | "links_into"        // file -> file or file -> module, derived from shallow build-file parsing (TRD §4.3), e.g. "compiled into the same target"
  | "member_of"          // any node -> feature node (AI classification result)
```

## 3. Flaw Flags (AI-assisted, always explicitly tagged as heuristic)

```ts
interface FlawFlag {
  id: string;
  kind: "dead_code" | "circular_dependency" | "structural_smell" | "suspected_bug" | "missing_error_handling" | "resource_leak_pattern";
  severity: "info" | "warning" | "high";
  description: string;          // human-readable, generated by the AI flaw-flagging task (TRD §5.2 item 3)
  confidence: number;            // 0.0–1.0, model-reported or heuristically assigned
  basis: "structural" | "ai" | "structural+ai"; // dead_code in particular should usually be "structural+ai" per TRD §5.2
  related_node_ids?: string[];   // e.g., the other side of a circular dependency
}
```

**Rule:** `dead_code` flaws must never be generated from AI judgment alone. They must always be gated on the structural fact `has_incoming_calls == false` (or, for files, "no node in this file is referenced anywhere") AND not flagged as `is_entry_point_candidate`. The AI's role for dead code is specifically to look at that structural fact plus context and assess plausibility (e.g., "this might be a callback registered via a function pointer table, not truly dead" vs. "this really does look unused") — never to flag dead code purely from reading code in isolation. This rule directly implements TRD §5.2 item 3 and must be enforced in code, not just convention.

## 4. Node ID Stability

Node IDs must be **stable across re-runs** when the underlying code hasn't meaningfully moved, so that:
- Incremental re-mapping (TRD §3.3) can correctly diff "same node, unchanged" vs. "same node, changed" vs. "genuinely new node."
- AI annotations persist across re-runs instead of being regenerated needlessly.
- The future diff/compare feature (PRD §4.2.12) has something stable to diff against.

**ID formula:** `sha256(file_path + ":" + node_type + ":" + qualified_name)` truncated to a fixed-length hex string, where `qualified_name` includes enclosing scope (e.g., module path, enclosing struct/impl for a method) so that two functions with the same short name in different scopes never collide. Line numbers are deliberately **excluded** from the ID so that adding a blank line above a function does not change its ID (only its `span` updates). If a function is genuinely renamed, it is correctly treated as a new node with a new ID — this is intentional and acceptable; rename-tracking is out of scope for v1.

## 5. Control Flow Graph (for execution-order reconstruction)

```ts
interface ControlFlowGraph {
  entry_block_id: string;
  blocks: ControlFlowBlock[];
}

interface ControlFlowBlock {
  id: string;
  kind: "sequential" | "branch" | "loop" | "return";
  calls_in_block: { edge_id: string; order_hint: number }[]; // calls made in this block, in source order
  next_blocks: { block_id: string; condition?: string | null }[]; // branch targets; sequential blocks have exactly one with condition = null
}
```

This structure is what `/api/graph/trace/:id` (TRD §6.2) walks to produce an **ordered, branching** execution trace, rather than a flat unordered list of callees. The frontend's "show execution order" mode (PRD §4.1.5, App Flow doc) renders this as a numbered/branching sequence, explicitly showing where the path forks on a condition — this is the direct technical fulfillment of PRD's "what is the work order when that function is called when other is called."

**Explicit limitation to surface in the UI:** this is a *static* control-flow reconstruction. Where a branch condition is data-dependent (e.g., `if (user_input == 5)`), the UI must show **both** branches as possible paths, not pretend to know which one actually executes at runtime — labeled clearly (e.g., "Path A — if condition holds" / "Path B — otherwise"), consistent with TRD §3.2's stated non-goal of full data-flow evaluation.

## 6. SQLite Schema (storage-level realization of this model)

```sql
CREATE TABLE nodes (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  name TEXT NOT NULL,
  language TEXT,
  file_path TEXT,
  start_line INTEGER,
  end_line INTEGER,
  signature TEXT,
  extra_json TEXT NOT NULL,        -- type-specific fields not promoted to columns (e.g. control_flow, member_node_ids)
  ai_summary TEXT,
  ai_summary_status TEXT NOT NULL DEFAULT 'pending',
  ai_model_used TEXT,
  structural_hash TEXT NOT NULL,
  has_incoming_calls INTEGER,       -- nullable; only meaningful for type='function'
  is_entry_point_candidate INTEGER,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
CREATE INDEX idx_nodes_file_path ON nodes(file_path);
CREATE INDEX idx_nodes_type ON nodes(type);
CREATE INDEX idx_nodes_name ON nodes(name);

CREATE TABLE edges (
  id TEXT PRIMARY KEY,
  from_id TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
  to_id TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
  type TEXT NOT NULL,
  confidence TEXT NOT NULL,
  order_hint INTEGER,
  condition TEXT
);
CREATE INDEX idx_edges_from ON edges(from_id);
CREATE INDEX idx_edges_to ON edges(to_id);

CREATE TABLE flaws (
  id TEXT PRIMARY KEY,
  kind TEXT NOT NULL,
  severity TEXT NOT NULL,
  description TEXT NOT NULL,
  confidence REAL NOT NULL,
  basis TEXT NOT NULL,
  primary_node_id TEXT NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
  related_node_ids_json TEXT        -- JSON array
);
CREATE INDEX idx_flaws_node ON flaws(primary_node_id);

CREATE TABLE files_manifest (
  file_path TEXT PRIMARY KEY,
  content_hash TEXT NOT NULL,
  language TEXT,
  last_parsed_at TEXT NOT NULL,
  parse_status TEXT NOT NULL,
  parse_error TEXT
);
```

`extra_json` exists deliberately so that adding a new type-specific field (e.g., a future field on `FeatureNode`) does not require a schema migration for every minor model extension — only fields that need to be efficiently filtered/sorted/indexed get promoted to real columns. New indexable fields **do** require a migration; this is intentional and acceptable.

## 7. JSON-over-the-wire Contract

All REST API responses (TRD §6.2) return JSON shapes that are **directly** the TypeScript interfaces defined in §1–§3 of this document (serialized via `serde` on the Rust side with `#[serde(rename_all = "snake_case")]` to match the field names exactly as written here). The frontend's TypeScript types must be hand-kept in sync with these interfaces (or generated from the Rust types via `ts-rs` / `specta` if the implementer prefers — either approach is acceptable, but the two sides must never silently drift apart). This document's interfaces are authoritative; if Rust structs and these interfaces ever disagree, this document wins and the code is fixed to match.
